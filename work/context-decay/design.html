Context Budget & Session Rollover

Measure any agent session's live context usage exactly from disk, warn before the ~150K-token "dumb zone", and roll work over to a fresh session via a deliberate, pruned handoff — instead of uncontrolled automatic compaction.

ai-workspace-template · implemented 2026-07-22 · spec: work/context-decay/context-decay-spec.md · reference: docs/context-budget.md

Problem & goals

LLM quality degrades past ~150K context tokens regardless of advertised window size, and agents cannot see their own usage — the numbers live in the API response envelope, persisted only in the runtime's on-disk session artifacts. Left alone, sessions drift into the dumb zone and then auto-compact, discarding context by heuristic. Goals:

  • Measure — exact, reproducible token counts from disk for Claude Code, Codex, Copilot (VS Code + CLI), and Gemini CLI.
  • Warn — reach the developer and, where possible, the agent itself, mid-task, before the threshold.
  • Roll over — a deliberate handoff that decides what the next session loads (and what it must not reload).
  • Neutral — bash + jq + Markdown skills; vendor-specific code confined to thin adapters and wiring.

Design decisions

D1Measure from disk, never ask the model.

Why: usage lives in the API envelope the model never sees; self-estimates are bad and optimistic.

Consequence: the agent's role is inverted — it invokes measurement at checkpoints, it is never the source of the number.

D2Per-runtime adapters behind a fixed contract.

Why: session formats are undocumented internals; drift is expected.

Consequence: a format change breaks one discover/measure pair, never the skills, docs, or hook.

D3Absolute threshold, not %-of-window.

Why: a 936K Copilot window doesn't move the dumb zone.

Consequence: one checked-in env pair (context-budget.env), raised in one place; STOP must stay below the runtime's auto-compact trigger.

D4Estimate fallback, never "unsupported".

Why: a hard failure silently disables the safety net where it's needed most.

Consequence: bytes÷4 with method=estimate; ±25% is fine given the WARN→STOP margin.

D5Explicit register over newest-mtime discovery.

Why: concurrent sessions make heuristics ambiguous.

Consequence: precedence is --transcript > registration > discovery; the hook gets the exact path on stdin and bypasses both.

D6Hook speaks only on escalation, throttled, fails open.

Why: warnings consume the budget they protect; repetition trains agents to ignore them; a broken hook must never block work.

Consequence: one WARN + one STOP per session, ≥60s between checks, any error → exit 0.

D7Rollover is a Markdown skill reusing handoff.

Why: vendor-neutral, and the workspace already has a handoff convention (see the checkpoint skill).

Consequence: backward handoff.md split from forward, deliberately pruned next-session.md with a "Do NOT reload" section; specialized workflow state files stay authoritative.

D8Layered triggering.

Why: only Claude Code has an in-band hook; other runtimes need other paths to the agent or the human.

Consequence: hook + skill checkpoints + watcher + standing CONTEXT.md instruction overlap deliberately.

D9Append-only JSON ledger.

Why: every measurement is also a data point about token growth per workflow phase.

Consequence: the safety net doubles as research data (work/context-decay/context-ledger.jsonl).

Runtime flow (Claude Code, the fully-wired case)

  1. Session starts; the agent runs scripts/context-budget.sh register (per the CONTEXT.md standing instruction) — pins the transcript in .context-budget/session-claude.json.
  2. Work proceeds. At each unit boundary the agent runs record --label "…", appending to the ledger and reading the exit code.
  3. Independently, every tool call fires the PostToolUse hook; at most once per minute it measures the transcript it was handed on stdin.
  4. Usage crosses 120K → the hook prints CONTEXT BUDGET WARN to stderr with exit 2, which Claude Code feeds back to the agent mid-turn. Once only.
  5. The agent finishes the current unit and runs the session-rollover skill: record → reflect → flush → handoff.md → next-session.md → bootstrap prompt → record.
  6. The user starts a fresh session and pastes the bootstrap prompt; the new session reads next-session.md, runs register, and continues — at a fraction of the token load, past nothing it needed to keep.
  7. Had usage instead crossed 150K, the hook's CONTEXT BUDGET STOP allows finishing only the current atomic step. 150K < Claude Code's ~200K auto-compact, so rollover always gets its chance first.

Limitations

  • Copilot CLI adapter unverified; Gemini CLI estimate-only (telemetry follow-up).
  • Hook granularity: one check per minute — a single huge tool result can overshoot between checks.
  • --runtime auto with several concurrent runtimes needs register or an explicit --runtime.
  • Non-macOS: Copilot storage roots differ; watch notifications need notify-send instead of osascript.
