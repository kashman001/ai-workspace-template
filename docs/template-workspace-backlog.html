Maintaining this backlog

Treat this file as a lightweight issue tracker for the template itself. The rules:

  1. IDs are stable and severity-scoped. High, Medium, Low, Decision + a number (e.g. M6). Never renumber or reuse an ID — once assigned it's permanent, even after the item is resolved.
  2. Every finding carries a status badge: Open · Resolved · Decided · Won't fix.
  3. When you fix an item: flip its badge to Resolved, add a Fixed: line describing the change (and the commit/PR if handy), and add class="resolved" to its <div class="find …">.
  4. When you find a new issue: append a new finding card under the right severity heading with the next free ID, status Open, and the Evidence / Impact / Fix lines.
  5. Update the scorecard counts and the "Last updated" date in the header on every change.
  6. Deliveries from other sessions/clones follow the same rules. A template change pushed from a downstream session must carry its backlog update in the same push; the session that pulls a change lacking one adds the missing row/card as part of incorporating it (see L14).
  7. Commit convention: reference the ID, e.g. git commit -m "Fix M6: …", so the git log and this file stay in sync.
  8. Re-review trigger: after any structural change to the template (new config file, new doc, new agent runtime), do a fresh evidence pass and append what it surfaces.

Tip: the matching skill for turning these into real tracked issues is to-issues (see docs/recommended-tooling.md) if you'd rather move them to GitHub Issues.

M9 context-budget.sh register reused the previous session's artifact instead of re-discovering MediumResolvedscripts/context-budget.sh

Evidence — 2026-07-22, first fresh session after the D4 build: register at session start reported tokens=146794 status=WARN — the previous session's transcript. resolve_session() consulted the saved registry (.context-budget/session-claude.json) before discovery, so register — whose whole purpose is to bind the current session — inherited the prior binding and never re-discovered. Every session after the first would measure the wrong transcript until it happened to overtake the stale one in mtime.

Fixed: resolve_session() now skips the registry lookup when the command is register, forcing fresh discovery; check/record/watch still trust the registry. Verified live: re-running register in the same session bound the correct artifact (tokens=50856 status=OK) and a follow-up check read it back.

M10 Copilot VS Code discovery pins a stale sibling session by mtime, ignoring VSCODE_TARGET_SESSION_LOG MediumResolvedscripts/context-budget.sh

Evidence — 2026-07-23, live copilot-vscode session on a workspace built from this template: register reported a false STOP. copilot_vscode_discover() picks the newest-modified chatSessions/*.jsonl; at register time the current session's log wasn't flushed yet, so it grabbed a stale rollover session's 1.4 MB log instead. The Copilot terminal exports VSCODE_TARGET_SESSION_LOG — the live session log's path — which the script ignored. Nearly triggered a needless rollover; the session was actually at 25% (tokens=38388 status=OK).

Fixed: copilot_vscode_discover() now uses $VSCODE_TARGET_SESSION_LOG when set and the file exists, falling back to the newest-mtime scan otherwise (var unset, or pointing at a not-yet-created log). Verified all three branches: pinned discovery returns the exact artifact and token count; unset and dangling-path cases preserve the old behavior. Adapter table in docs/context-budget.md updated. Follow-up (2026-07-23, 7c29476, delivered from the origin copilot workspace): on current builds the var is a debug-logs/<id> directory, not the token file — discovery now derives the session id from its basename and pins chatSessions/<id>.jsonl deterministically; newest-mtime remains only the var-unset fallback.

M11 claude_discover races concurrent same-workspace sessions (newest-mtime proxy) MediumResolvedscripts/context-budget.sh

Evidence — 2026-07-23 review of all adapters after M10: claude_discover() takes the newest-modified *.jsonl in the project dir. Claude Code flushes per-turn so the live session usually wins, but two concurrent sessions in one workspace race, and the transcript basename is exactly the session id Claude Code already exports as $CLAUDE_CODE_SESSION_ID (verified live) — the heuristic was never necessary.

Fixed: pin <proj>/$CLAUDE_CODE_SESSION_ID.jsonl when the var is set and the file exists; newest-mtime fallback otherwise. Differential-verified in one shell call: with a stale transcript touched newest, unpinned register binds the stale file, pinned binds the live session.

M12 Gemini shared telemetry log serves the previous session's counts at register MediumResolvedscripts/context-budget.sh

Evidence — gemini_measure() reads the last input_token_count from the workspace-scoped, append-only .gemini/telemetry.log, which is shared by every Gemini session in the workspace. A new session's register, before its first response, reads the previous session's final (potentially near-threshold) count — a false WARN/STOP needing no concurrency at all. Concurrent sessions also interleave, attributing whichever response landed last.

Impact — same failure the M10 incident showed: a fresh session can be told to roll over immediately.

Fixed: register --runtime gemini now truncates the workspace telemetry log after binding it, safe under the single-session-per-runtime assumption (L12); measures then only ever see the current session's responses. When neither responses nor a chat log exist yet, gemini_measure reports 0 estimate (fresh session, empty context) instead of failing. Lifecycle-verified with fixtures: stale 140K count read as WARN pre-fix → register truncates and reports tokens=0 → appended response read back exact. Rejected alternative: filtering OTLP records by session id — Gemini CLI exports no session-id env var to match against. Live-session verification rides the existing Gemini-auth gated item.

L11 Codex & Copilot CLI discovery share the newest-mtime race; codex fallback can bind another project LowResolvedscripts/context-budget.sh

Landed 2026-07-23: the any-project codex fallback is removed — codex_discover() now fails ("no session artifact") rather than bind another project's rollout; verified on a machine with 4 non-matching rollouts (pre-fix would have bound the newest).

Fixed: codex now pins rollout-*-$CODEX_THREAD_ID.jsonl — live-verified on the origin machine (the env var was probed inside a real codex exec session, its value equals the rollout UUID and session_meta.payload.id, and the injection is confirmed in openai/codex source); differential-verified (a stale rollout touched newest still loses to the pinned live session). Copilot CLI now pins session-state/$COPILOT_AGENT_SESSION_ID/events.jsonl under ${COPILOT_HOME:-~/.copilot} and scans the correct session-state/ path (the unattested sessions/ path is dropped; legacy history-session-state/ kept as fallback) — research-sourced from the CLI changelog and fixture-verified, but live verification still pends an install, so the copilot-cli adapter stays marked unverified.

L12 Session registry is keyed per-runtime — concurrent same-runtime sessions clobber each other LowResolvedscripts/context-budget.sh · docs/context-budget.md

Evidence — register writes .context-budget/session-<runtime>.json; two concurrent sessions of the same runtime in one workspace overwrite each other's artifact pin, so check/record/watch in one session can measure the other's transcript even after M10/M11-style discovery pins.

Fixed: documented single-session-per-runtime-per-workspace as an explicit limitation in docs/context-budget.md (the M12 reset-at-register fix leans on the same assumption; concurrent same-runtime sessions belong in separate workspace clones). Rejected alternative: keying the registry by session id — the in-band hook and ad-hoc check calls would then need the session id to find their registry entry, complexity the single-session reality doesn't justify. Revisit only if concurrent same-runtime sessions become a real workflow.

L13 Session registration is documented for agents only — no developer-facing lifecycle doc, no mechanical trigger LowResolveddocs/context-budget.md · .claude/settings.json.example

Evidence — the developer quickstart in docs/context-budget.md shows check/watch but never says sessions are registered at start; register appears only in the agent quickstart and CONTEXT.md standing instruction. Registration is by-instruction, not by-mechanism: no SessionStart hook exists, so an agent that ignores its context file never registers.

Impact — a developer reading only their quickstart can't tell whether registration is automatic or their job; an instruction-ignoring agent silently depends on discovery fallbacks.

Fixed: session-lifecycle paragraph added to the developer quickstart (registration is the agent's job, automatic-by-instruction via CONTEXT.md, only for sessions started inside the workspace tree; unregistered sessions fall back to discovery), and a SessionStart hook added to .claude/settings.json.example that runs register --transcript <path from the hook payload>, making registration mechanical for Claude Code. Hook event live-verified (this session received SessionStart hook context); hook command verified end-to-end against a simulated payload (pins the exact transcript, exits 0) and fails open on an empty payload.

L14 Template changes delivered from downstream sessions bypass the backlog convention LowResolvedCONTEXT.md · this file

Evidence — three same-day deliveries pushed from downstream sessions (7c29476 copilot-vscode pin follow-up, bd40023 launcher/ledger convention, f3928e6 diff-review.sh) each landed with no backlog row or card update; each had to be reconciled after the fact by the pulling session (df12f76, 2ed1eec, eeb0a78).

Fixed: the convention now covers deliveries explicitly, on both ends — a new rule in the "Maintaining this backlog" list (delivering commit carries its backlog update; the pulling session adds any missing row as part of incorporation) and a matching sentence in the Template Backlog section of CONTEXT.md, which downstream template-repo sessions load.
