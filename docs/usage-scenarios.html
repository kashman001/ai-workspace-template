00 · How to read this

The workspace is a coordination layer — one shared context for Claude Code, Codex, OpenCode, and Gemini — not product code itself. The journey below has a fixed spine and one fork:

  • Pull & bootstrap the template into your own workspace (every project does this once).
  • Then the fork: Branch A pull in an existing repo, or Branch B start a new project from scratch.
  • Build features in a repeatable loop with AI agents, persisting state to work/.
  • Checkpoint at session boundaries so the next (post-compaction) session resumes cleanly.

Each scenario names the human action, the agent role, and the concrete artifacts touched. Throughout, the rule is the same: disk is the source of truth — durable state lives in files under docs/ and work/, never only in an agent's conversation memory.

02 · Scenario 1 — Pull the template & bootstrap

S1Fresh clone → working workspace

Human drives setup · Agent (optional) fills placeholders from the Agent Bootstrap Instructions in docs/workspace-structure.md

  1. Click “Use this template” on GitHub (or clone and reset history), then enter the new directory.
  2. Run the bootstrap script — it creates .env, the agent-entrypoint symlinks, and repos/README.md: 
    git clone https://github.com/<you>/ai-workspace-template.git <project>-workspace
cd <project>-workspace && rm -rf .git && git init
./scripts/setup.sh
  3. Fill the <…> placeholders — CONTEXT.md first (the front door), then README.md, SPEC.md, docs/repos-registry.md. Find what's left with grep -rIn -e '<[a-z-]\+>' -e 'TODO' -e 'Fill in:' .
  4. Wire agents & MCP: copy .claude/settings.json.example → .claude/settings.local.json, .mcp.json.example → .mcp.json, export the GitHub token, and install yt-dlp if you want the workspace-local YouTube transcript server (docs/mcp-setup.md).
  5. Verify with the scripts/check-*.sh checks; fix anything they flag using the matching runbook in docs/runbooks/.

Exit state: Workspace bootstrapped — every agent reads the same CONTEXT.md; you now decide the fork below.

03 · Scenario 2 — Branch A: pull in an existing repo

S2Branch A  Onboard an existing product repo

Human clones the repo · Agent runs the onboard-repo skill to index it and derive context docs

  1. Clone the product repo under repos/ (multi-repo) — it's gitignored, so it stays out of the workspace's own history.
  2. Onboard it so agents understand it without re-reading the whole tree each time: 
    /onboard-repo <repo-name>        # Claude Code shortcut
# or: scripts/onboard-repo.sh <repo-name>
    This records identity/auth in docs/repos-registry.md, builds a graphify code index (manual-scan fallback if graphify is absent), and writes code-structure.md / design.md / api.md under docs/repo-context/<repo>/.
  3. Skim the generated docs/repo-context/<repo>/ docs and correct anything the scan got wrong — these become the agent's map of the codebase.
  4. Keep it fresh: scripts/check-repo-context.sh flags drift; scripts/onboard-repo.sh <repo> --refresh regenerates.

Exit state: Existing repo onboarded — registry + per-repo context docs + (optionally) a graphify graph. Re-run for each additional repo. → joins the development loop.

04 · Scenario 3 — Branch B: start a new project

S3Branch B  Scaffold a brand-new project

Human sets product intent · Agent brainstorms scope and scaffolds the first slice

  1. Decide the shape: a single-repo workspace (product code lives at the workspace root, repos/ stays empty) or a new repo cloned under repos/.
  2. Write product intent into SPEC.md and the one-paragraph purpose in CONTEXT.md → Workspace Purpose.
  3. Pressure-test the idea before coding — invoke the brainstorming skill, then grill-with-docs to align terminology against the project's Language section.
  4. Turn the agreed plan into tracked work with to-prd (a PRD) and to-issues (tracer-bullet vertical slices).
  5. Scaffold the first runnable slice — the smallest end-to-end path that proves the architecture.

Exit state: New project scaffolded — spec, language, and a first slice of issues exist. → joins the development loop.

The fork re-converges

Branch A and Branch B differ only in how the code arrives. Both land in the same place: a workspace where agents have a reliable map of the code (repo-context docs or a fresh SPEC) and a queue of vertical slices to build. Everything after this point is identical.

05 · Scenario 4 — The feature loop with AI agents

The core working rhythm: one vertical slice at a time, red-green-refactor, with agents driving each phase and durable state written to work/<username>_<project>/.

Figure 2 — One slice through the loop. Repeat until the feature is done.

S4Build one slice with agents

Human sets the goal & reviews · Agent runs the loop: brainstorm → TDD → self-review

  1. Pick the next issue/slice. Restate it as a verifiable goal (“write a failing test for X, then make it pass”) — strong success criteria let the agent work independently.
  2. Red: agent writes a failing test that reproduces the requirement (test-driven-development skill).
  3. Green: minimum code to pass — no speculative features, surgical changes only.
  4. Refactor with tests green; then requesting-code-review before merge.
  5. After modifying code, run graphify update . to keep the knowledge graph current (AST-only, no API cost).
  6. Persist anything that must survive context compaction to work/<username>_<project>/ — not to conversation memory.
  7. When you pick one approach over a real alternative, capture the why (decision-log skill / /decision): a one-line Decision: commit trailer, and a note in work/<user>_<project>/decisions.md — code records what, not how it got there. Lasting-weight calls get promoted to an ADR at checkpoint.

Loop: repeat per slice. When the feature's tests all pass → finish the branch (Scenario 6). At a session boundary → checkpoint (Scenario 5).

Context discipline keeps long sessions healthy

Disk is the source of truth — re-read files before asserting facts; demand-load rather than pre-load; use targeted Grep/ranged reads; treat sub-agent summaries as hints to verify. These rules (in CONTEXT.md → Agent Context Discipline) are what let the loop run for hours without drifting.

06 · Scenario 5 — Checkpoint & resume across sessions

S5Wrap a chunk, hand off, pick up clean

Human calls the checkpoint · Agent reconciles state and writes the hand-off

  1. At a natural boundary (or before compacting context) invoke /checkpoint [next-focus].
  2. The skill reconciles the backlog / issue tracker, project memory (+ MEMORY.md pointers), and reference docs — bringing each in line with what actually shipped. It also scans work/<user>_<project>/decisions.md for notes flagged Promote?: yes and promotes them to committed ADRs under docs/adr/ (the decision-log scheme) — ephemeral why becoming durable record before context compacts.
  3. It writes a secret-free hand-off doc (via the handoff skill) under work/<username>_<project>/, referencing specs/plans by path rather than duplicating them.
  4. It confirms a clean branch state and emits a catch-up prompt for the next session — naming the hand-off doc path so the post-compaction agent knows where to start.
  5. Next session: paste the catch-up prompt → the agent reads the hand-off doc from work/ and resumes the development loop.

Exit state: back into the Active development loop with no lost context.

07 · Scenario 6 — Finish a branch

S6Feature complete → merge → next

Human approves the merge · Agent verifies completion before claiming done

  1. All slice tests pass and the feature meets its success criteria — verify by running the commands, not by assertion (verification-before-completion).
  2. Use finishing-a-development-branch to choose the integration path (merge, PR, or cleanup).
  3. Open the PR against main; address review; merge.
  4. Run graphify update . on the merged state, then return to the loop for the next feature.

Exit state: Branch merged → loop back to the next slice, or checkpoint and stop for the session.
